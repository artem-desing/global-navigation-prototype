# 05 — QA stress test (functional reality check)

**Lane:** functional QA. **Scope:** v0, v2, v3, v4. **v5 excluded.**
**Date:** 2026-04-30. **Tester:** QA agent.
**Build:** local `pnpm build` against `main` (commit `43bd46c`).
**Live deploy:** `https://artem-desing.github.io/global-navigation-prototype/v/{v0,v2,v3,v4}/`.
**Browsers exercised:** static-export inspection + curl HEAD checks against the deployed URL. WebFetch was denied at runtime, so no in-browser DOM inspection of the deploy. All click-through behavior validated by reading source against the manifest and the build output.

## 1. Executive summary

| | Variant |
|--|--|
| **Fewest functional defects** | **v0** — only one in-scope defect (no inline). It carries the framework defects (embargo, AI panel esc, no responsive collapse) but adds nothing of its own. |
| **Most functional defects** | **v4** — most surface area (collapsed rail + expanded rail + hover menus + persistent SecondColumn rule + ⌘B keybind). Carries five distinct defects beyond the shared framework set. |
| **Biggest single risk** | **AI Hypervisor visible in every variant's chrome and home page.** Per `CLAUDE.md` embargo, this string must not appear in user-visible artifacts pre-2026-06-04. Deployed to a public GitHub Pages URL today. |

All four variants build cleanly under `output: 'export'`, render their basePath-prefixed routes, and produce a static `404.html` for invalid URLs. ⌘K is window-listener-driven and works equivalently across all four. No Critical defects in routing, drill, or basePath integrity. Theme parity reads correctly on inspection — surface-1/border tokens and `--color-bg-light-primary` hover are used uniformly; no surface-N stacking traps in any in-scope variant.

The shared framework defects (embargo strings, AI assistant panel has no Escape handler, no responsive collapse below 1024) hit all four equally.

## 2. Per-variant findings

### v0 — Always-open sidebar

**Routing.** Direct-load works for every probed manifest URL (200 across `/v/v0/`, `/v/v0/edge/data-planes/`, `/v/v0/edge/data-planes/production/`, `/v/v0/edge/data-planes/production/alerts/`). Build output confirms full tree under `out/v/v0/`. Invalid product ID returns the `mode: 'unknown'` `<NotFound>` UI from `catch-all-client.tsx:143`. Invalid top-level paths return GH Pages' 404.html.

**Keyboard.** Native Tab traverses Home → Recent → product items → utility items. No arrow-key handler on the rail (v0 doesn't define one — it's a column of links, native Tab is enough). No Escape on the rail itself. Search dialog opens via window-level ⌘K from any context.

**Dark/light.** Rail uses `--color-bg-surface-1`; hover overlays use `--color-bg-light-primary`; scope chips use `--color-bg-light-primary` against `surface-1`. No surface-2/3/4 stacking. Border-primary-light separators give visible delineation in light mode. Theme parity is correct on inspection.

**Narrow widths.** **Fails at 1024 and below.** `Rail` is `w-96` (96px under WADS' 1px spacing) + `SecondColumn` is `w-[256px]` = **352px of fixed chrome** before main. At 1024 leaves 672px for content (workable). At 375 the layout breaks — there is no media query. Acceptable per prototype scope, document this as Medium.

**Embargo.** "AI Hypervisor" visible in rail (manifest `label`/`shortLabel`), in breadcrumb, in second column header, and in the home `SummaryCard` title.

**WADS scans.** No barrel imports. No third-party icon imports. No `// @ts-ignore` / `// eslint-disable`. Hardcoded hex: `#FF441C` in `wordmark.tsx:25` — this is the brand-orange chevron of the Wallarm wordmark SVG; it stays the same across themes by design. Acceptable.

**Regressions.** None observed against v0's prior fixes — v0 is the baseline.

**Build.** Compiles. 2919 static pages emitted; v0 tree complete in `out/v/v0/`.

### v2 — Hover-to-expand icon rail

**Routing.** All probed paths 200 on the deploy; `/v/v2/edge/data-planes/production/` ✓, `/v/v2/ai-hypervisor/heatmap/` ✓. Variant-equivalent paths render in v2 the same way they do in v0 (the v2 shell reuses v0's `SecondColumn`, `Breadcrumb`, `TopBar`).

**Keyboard.** `handleRailKeyDown` at `v2/rail.tsx:209` handles ArrowDown/ArrowUp and Escape. Escape collapses, unpins, and blurs out — the only variant with explicit rail-Escape. Roving-focus is via `itemRefs` array, populated during render via `assignRef`.
- React 19 lint flags this pattern (`itemRefs.current = ...slice()` during render, mutating `refIndex` outside callback) but it works in practice. Concern, not Critical.

**Dark/light.** Same surface-1/border-primary-light scheme as v0. Pin button uses `light-primary` for active and hover backgrounds — visible against surface-1 in light mode. No surface-stacking issues observed.

**Narrow widths.** Rail collapses to 64px when not pinned, expands to 192px on hover. Plus SecondColumn 256px, that's 320px chrome at hover-rest, 448px while expanded. Same caveats as v0 — no media-query collapse. Slightly better at 1024 because rail collapses to 64.

**Embargo.** Same as v0 (shared TopBar + SecondColumn).

**WADS scans.** No barrel imports. No third-party icons. No ts-ignore / eslint-disable in v2. Hardcoded hex: none in v2 source.

**Regressions.** Memory `project_v2_icon_rail` calls out three load-bearing fixes:
- `onNavigate` chain → present at `v2/rail.tsx:343,352,368,392`.
- Focus-suppression after navigation → present (`suppressFocusInRef` at line 125, used in `handleFocusIn` at 174).
- Unpin-blur (`togglePin` blurs activeElement on unpin) → present at lines 271–276.
All three holding.

**Build.** Compiles. v2 tree complete in `out/v/v2/`.

### v3 — Icons-only with tooltips

**Routing.** Deep links 200 on the deploy. The shell re-uses v0's TopBar + SecondColumn + Breadcrumb (`shell.tsx:5-10`), so the route surface is identical to v0. Recent dropdown and utility dropdowns render via WADS DropdownMenu.

**Keyboard.** `handleRailKeyDown` (`v3/rail.tsx:62`) handles only ArrowDown/ArrowUp — **no Escape handler**. Native Tab through items works. Custom `RailTooltip` (lines 161–216) opens on `onFocusCapture` so keyboard users get tooltips too — that's correct.
- The `RailTooltip` component's tooltip span is a sibling div inside a wrapper — `aria-describedby` is **not** set on the inner trigger, so screen readers announce the tooltip only via the unhighlighted role="tooltip" on the absolutely-positioned span. WCAG-soft, not a hard fail.

**Dark/light.** Same surface tokens as v0/v2. Tooltip uses `--color-component-tooltip-bg` / `--color-text-primary-alt-fixed` — those are WADS-named tooltip tokens, theme-aware.

**Narrow widths.** Rail is fixed 64px. SecondColumn 256px. Total 320px before main. Best of the four for narrow widths but still no <375 fallback.

**Embargo.** Same as v0/v2 (shared TopBar + SecondColumn + manifest labels).

**WADS scans.** No barrel imports. No third-party icons. No ts-ignore. Hardcoded hex: none.

**Regressions.** v3 has no expand/collapse, so the v2 onNavigate / focus-suppression / unpin-blur axes don't apply. It does carry the same React-19 lint pattern (refIndex / itemRefs.current mutation during render) — 4 lint errors at `v3/rail.tsx:60,82,101`. Not user-facing.

**Build.** Compiles. v3 tree complete in `out/v/v3/`.

### v4 — Pop-out menus, ⌘B merged sidebar

**Routing.** Deep links 200 on the deploy. v4's `shell.tsx:27-29` chooses to render the v0 SecondColumn only on `/settings/*` or when `ctx.backHref !== null` (drilled scope). Verified the rule in source. Tested URL `/v/v4/edge/data-planes/production/` returns 200; `/v/v4/testing/test-suites/` returns 200.

**Keyboard.** Two keybinds: window-level ⌘K (search) and window-level ⌘B (toggle expand). They use different keys — no collision. ⌘B has `e.preventDefault()` so it won't trigger browser bookmarks bar or bold (`v4/rail.tsx:67`). The collapsed rail's ArrowDown/ArrowUp roving works (line 107). No Escape handler.
- `expanded-rail.tsx` (the merged sidebar) has **no `onKeyDown` handler at all**. ArrowDown/ArrowUp don't navigate items in expanded mode, only Tab. Inconsistent with collapsed mode.
- Hover-menu in collapsed mode (`ProductRailItem` at line 349) opens on `onMouseEnter` (120ms delay) and on `onFocus` (immediately). Closing via blur is gated on whether the new focus target is inside the wrapper. **No Escape to close** the hover menu.

**Dark/light.** Hover menu (`SectionHoverMenu`, line 488) uses `--color-bg-surface-1` background, `--color-border-primary` border, and `shadow-lg`. Border + shadow give a visible boundary against the rail's `surface-1` background → no white-on-white. The expanded-rail's tree uses the same hover token (`light-primary`).

**Narrow widths.** Collapsed rail 64px; SecondColumn shows only when drilled / on `/settings/*`. So unscoped product views have **just 64px of chrome** in v4 — best for narrow widths among the four. Hover menu at 240–320px appears next to the rail, can collide at narrow widths but doesn't reflow the layout.

**Embargo.** Same as the others. The hover menu also displays "AI Hypervisor" (the `manifest.shortLabel ?? manifest.label` at `expanded-rail.tsx:258`).

**WADS scans.** No barrel imports. No third-party icons. No ts-ignore. Hardcoded hex: none in v4 source.
- Unused `Manifest` import at `v4/expanded-rail.tsx:32`. Low.

**Regressions / specific concerns.**
- `useExpandedRail` (`expand-state.ts:39-41`) calls `setExpanded(readExpanded(slug))` directly inside a `useEffect` after init-from-localStorage. Lint flags this — second-render flash if the `slug` prop ever changes mid-mount. Not currently observable, but a latent bug.
- v4's expanded-rail does not honor the v4 SecondColumn delegation rule (memory `project_v4_second_column_delegation`) — it always shows the full top-level tree of every product side-by-side. That's the design intent. But when the user is *also* drilled into a scope, the expanded rail shows the unscoped tree at the same time as the SecondColumn shows the scoped tree. Two simultaneous trees could confuse — flag as Medium.
- The `growUpward` prop of `ProductRailItem` and `SectionHoverMenu` is declared (lines 346, 473) but never set true by any caller. Dead parameter — Low, hygiene.

**Build.** Compiles. v4 tree complete in `out/v/v4/`.

## 5. QA gate

| Variant | Stakeholder-share-ready as a prototype? | Conditions |
|---|---|---|
| **v0** | **Yes**, with caveats. | Caveat: embargo strings visible (per Artem's call). 375px is broken. AI panel Escape gap is a known cross-cutting issue. |
| **v2** | **Yes**, with caveats. | Same shared caveats as v0. v2's three load-bearing fixes are intact. Lint warnings are not user-facing. |
| **v3** | **Yes, but…** Tooltip-only rail leans hard on hover discoverability. Escape on rail missing (Defect 5). | Recommend adding rail-Escape (parity with v2) before sharing — keyboard users currently have no way to "step off" the rail without Tab-out. |
| **v4** | **Conditional.** | The dual-tree state (Defect 8 — expanded rail + scoped SecondColumn rendering simultaneously) is the most visible behavior gap among the four. Recommend resolving before stakeholder review, or flagging the duplication explicitly in the demo. |

**Cross-variant blocker for any external sharing:** the AI assistant panel has no Escape handler (Defect 2). This affects all four equally and is a genuine a11y regression. Fix once, fixes everywhere.

**Single recommendation for the next iteration:** before adding a v6 / v7, harvest the keyboard-handling gaps from this report. v2 has the strongest keyboard story (Escape, ArrowKeys, focus suppression, unpin-blur). v3 and v4 should match that bar before they're treated as peers.
