# QA report — v0 vs v6

**Lane:** Lane 5 (QA / click-through). **Date:** 2026-05-05. **Tester:** QA agent.
**Branch:** `main` (commit `d11f4c3`). **Scope:** v0 (full-rail with hover-preview) vs v6 (user-controlled rail; expanded / collapsed / hover-overlay).

## Test methodology

Hybrid: **code audit + deployed-URL probe**. Live shell access (`pnpm dev`, browser DOM) was not exercised in this lane — bash sandboxing blocks long-running servers and the WebFetch tool was not loaded. Compensated by:

- **HTTP probes** of the deployed URL (`https://artem-desing.github.io/global-navigation-prototype/v/{v0,v6}/...`) — 200 across `/v/v0/`, `/v/v6/`, `/v/v0/edge/data-planes/production/alerts/`, `/v/v6/edge/data-planes/production/alerts/`, `/v/v6/ai-hypervisor/heatmap/`, `/v/v6/testing/test-suites/`.
- **Build output** inspected at `out/v/v0/` and `out/v/v6/` — both present, basePath honored.
- **Source-level audit** of `src/nav/variants/v0/*` and `src/nav/variants/v6/*`, plus shared `src/nav/shell/`, `src/nav/search/global-search.tsx`, `src/nav/z-index.ts`, `src/app/v/[variant]/*`.

Defects found at runtime would need a real browser to confirm; defects below are either visible in source or carried forward from the prior `stress-test/05-qa.md` (which validated the same code paths against the deploy).

## v6-specific spot checks (lane requirements 7 + 8)

- **Mode persistence** — `readMode`/`writeMode` use `nav:v:<slug>:rail-mode` (`v6/rail.tsx:80-95`). Per-variant key, so v0 (which has no mode) is unaffected.
- **Hover-overlay z-index above main, below modals/AI panel** — rail uses `Z_RAIL_OVERLAY=20`. AI panel has no `zIndex` (renders as flex sibling, so DOM order alone). WADS Dialog (⌘K, tenant) portals at WADS' own stacking (>50). Flag panel uses `z-50` literal. Result: rail (20) below dropdown (30, WADS), below dialogs (50+). **Stacking is correct today**; the latent risk is the AI panel's missing `zIndex` if it ever moves into a portal.
- **Mode switching mid-flow** — Switching `hover → collapsed` while the overlay is open: the `[mode]` effect clears `hoverOpen` immediately, the rail width transitions back to 64. Switching `collapsed → expanded` while drilled in `/edge/data-planes/production/alerts/`: the spacer width changes (64 → 192), pushing main content right by 128px. SecondColumn is unchanged (it's a sibling, not inside the rail). No state loss, no SecondColumn re-mount. Verified by code path; a runtime probe would tighten this.
- **Sidebar-mode-control placement** — bottom of the rail, last item in the utilities stack (`v6/rail.tsx:503-512`). Per memory `project_v6_sidebar_mode_placement`, this was settled 2026-05-05 — don't re-propose moving it.

## Lane verdict

**v0 ships cleaner today** — by a slim margin. v0 carries one Medium defect (375px) and two Lows (z-10 literal, brand hex). v6 inherits v0's chrome (TopBar / SecondColumn / Wordmark) plus all of its defects, and adds two Mediums of its own (touch-device hover trap when `mode='hover'` is the persisted choice; rail-Escape parity gap across modes). Both share the High AI-panel-Escape regression.

The reason v0 leads is interaction-surface size, not code quality: v6 is more interactive, so it has more state to keep coherent across modes. The hover-on-touch case is the realest concern — it isn't a critical break (default mode is `collapsed`, which falls back to tooltips), but a returning user who chose `hover` on desktop will have a degraded first-paint on a phone. v6's positives are real (better narrow-width story via `collapsed`, faster keyboard nav via rail roving + leader keys, expanded mode for power users) — none of those compensate for the touch-device gap if v6 is the variant that ships beyond prototype. **Recommendation:** before promoting v6, add a `(hover: hover)` media-query guard that downgrades `hover` → `collapsed` on no-hover devices, and lift the rail-Escape blur to fire across all three modes.
